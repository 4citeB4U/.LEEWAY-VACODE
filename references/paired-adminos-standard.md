# LeeWay Paired AdminOS Standard

## Doctrine
LEEWAY PAIRED APPLICATION DOCTRINE:

A LeeWay application is never just the public app.
A LeeWay application includes the public app and the owner’s governed AdminOS.

The public app serves the world.
The AdminOS serves the owner.
The agents serve under law.
The database stores state but does not grant authority.
The owner controls the experience.
LeeWay controls the rules.
Nothing becomes live without governed projection and human approval.

## Required paired output
Every LeeWay app creation should produce:
1. Public-facing application
2. AdminOS control plane
3. Draft/published projection model
4. Owner manual and onboarding
5. Agent workforce
6. Agent skills and proposal runtime
7. Runtime authority model
8. Audit and telemetry
9. Security/privacy layer
10. MCP/tool registry
11. Compliance checker
12. Deployment/runtime validation reports

## Completion rule
A LeeWay app is not complete when the public interface renders.
A LeeWay app is complete only when the owner can govern it.

## AdminOS requirements
The AdminOS must allow the authorized owner to:
- view the public app
- edit controlled content
- manage products/data where applicable
- manage media
- manage theme/settings
- inspect agents
- inspect MCP/tools
- inspect runtime authority
- inspect audit and telemetry
- use guided onboarding
- print/download the owner manual
- ask agents for help
- approve/reject agent proposals
- publish manually
- freeze unsafe activity when needed

## Projection contract
Public app reads published state.
AdminOS edits draft state.
Draft preview reads draft state.
Published preview reads published state.
Publish requires validation and owner action.
Agents apply only to draft after approval.
Public app never writes AdminOS authority state directly.

## Violation
If a public app is created without a paired AdminOS, LeeWay audit must flag it unless a governed static-only exception exists.
